## Project Goal

Build an MVP prototype that replaces traditional Letter of Credit (L/C) and Bill of Lading (B/L) workflows with a trustless smart escrow system.

The system should release funds automatically when logistics telemetry and customs data confirm shipment conditions.

Core verification data:
- VGM confirmation from port IoT
- AIS vessel departure status
- CEISA / customs approval
- e-BL CID validity from IPFS

The MVP must run locally first. Use mock data and mock services when real API keys are unavailable.

## Tech Stack

- Blockchain: Local Hardhat network first, then Polygon Amoy testnet later
- Smart Contract: Solidity ^0.8.20 + Hardhat
- Oracle MVP: Trusted backend oracle signer using Node.js
- Future Oracle Direction: Chainlink Functions / Chainlink Automation / multi-oracle consensus
- Storage: IPFS concept using mock upload first; structure code so web3.storage can be added later
- Backend: Node.js + Express for oracle gateway and API aggregator
- Frontend: React + Ethers.js + TailwindCSS
- Wallet abstraction: ERC-4337 should be represented as a planned/future module only. Do not fully implement ERC-4337 unless the core MVP is already working.
- Testing: Hardhat test suite

# What to Build

## 1. Smart Contract

Create:

/contracts/SternEscrow.sol

Implement a state machine:

Pending -> Verified -> Completed
Pending -> Refunded
Pending -> Disputed
Disputed -> Completed
Disputed -> Refunded

Escrow logic:
- Importer locks funds when creating escrow.
- Store:
  - contractValue
  - exporterAddress
  - importerAddress
  - arbiterAddress
  - eBLCID
  - deadline
  - createdBlock
  - commodity
  - containerRef
- Oracle submits verification data:
  - VGM match
  - AIS departed
  - CEISA approved
  - e-BL CID valid
- Release condition:
  IF VGM_match == true
  AND AIS_departed == true
  AND CEISA_approved == true
  AND eBLCID_valid == true
  AND at least 128 blocks have passed since creation
  THEN transfer funds to exporter and emit eBL transfer event.

Security requirements:
- Use ReentrancyGuard.
- Use Ownable or role-based access for trusted oracle.
- Use Pausable.
- Validate addresses.
- Prevent double release.
- Prevent refund after completed.
- Allow importer refund only after deadline if escrow is not completed.
- Emit events for escrow creation, oracle verification, fund release, refund, dispute opened, dispute resolved, and e-BL transfer.

Dispute logic:
- Any party can open dispute before completion.
- Use 2-of-3 approval concept between exporter, importer, and arbiter.
- Two approvals for manual release -> transfer funds to exporter.
- Two approvals for refund -> refund importer.
- Prevent the same address from voting twice.

## 2. Hardhat Project

Create:
- hardhat.config.js or hardhat.config.ts
- scripts/deploy.js
- test/SternEscrow.test.js
- .env.example

Tests must cover:
1. Happy path:
   - importer creates escrow with funds
   - oracle submits all true
   - after enough block confirmations, funds release to exporter
2. Invalid oracle:
   - non-oracle cannot submit verification
3. Mismatch path:
   - VGM false prevents automatic release
4. Dispute path:
   - VGM mismatch
   - dispute opened
   - 2-of-3 approvals trigger refund
5. Timeout path:
   - deadline passed
   - importer can claim refund
6. Reentrancy / double execution:
   - second release attempt must revert

For testing 128 block confirmations, use Hardhat helpers or mine blocks manually.
